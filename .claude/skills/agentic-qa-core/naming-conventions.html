The Naming Codex

Every test artifact in this framework has one exact name shape. This is the single source of truth — from a TMS test case to a KATA component, a branch, a folder. Learn the shape once; read any artifact's intent at a glance.

Before the shapes — why this system exists

Why a naming codex?

A name is not decoration: it is the first datum of every artifact. When each name follows a strict shape, its layer, its scope and its intent read without opening anything — by a human or a machine.

Readable at a glance

The prefix and first token say what it is and which layer it belongs to before you read the sentence.

Traceable

The Jira key travels inside the name, so Story ↔ test ↔ code ↔ branch stay tied together.

Machine-readable

Lint, sync and the kata-manifest rely on predictable shapes to automate.

Anti-duplication

One shape per artifact prevents two names for the same thing.

Learn the shape once; read any artifact's intent forever.

Vocabulary first — define this before any name

Core concepts

The whole codex uses these terms. Read them once; the rest of the deck assumes you already know them.

TMS

Test Management System — where tests, plans and runs live (here, Jira + Xray).

Epic / Story / PBI

Epic = feature; Story = unit of work with acceptance criteria; PBI = the backlog item.

Test Case (TC)

One individual case — a single asserted behavior. The leaf of the tree.

Test Set

A logical grouping of TCs by feature. It neither plans nor runs — it only groups.

Test Plan

What will be tested and its scope (Story / Feature / Sprint / Product).

Test Execution

One run of that plan — the record of what passed or failed.

Precondition (PRC)

The state required before running (e.g. an authenticated user).

ATC

Acceptance Test Case — the KATA method that implements a TC in code.

KATA

The framework's layered test architecture (L1–L4). See the CODE layer.

Fixture

KATA's dependency injection — hands ready-made api/ui/test to the test.

Xray

The TMS plugin on top of Jira providing Test, Test Set, Test Plan, Test Execution.

Modality

How the project stores TCs: Xray (dedicated issues) or Jira-native (fields).

The map · names are organized by altitude

Seven Layers of Naming

A name's layer tells you what kind of thing it is before you read a word. Each layer owns a color in this codex.

CASEThe individual Test Case (TC) — what is assertedshould <outcome>
GROUPThe container of cases — the feature under testValidate <feature>
CONTAINERPlanning & execution vessels — ATP/ATR/FTP/STP + Test Set/PRC{ACRONYM}: {scope}: …
CODEKATA — components, ATCs, files, tags@atc · {Resource}Api
JIRAWorkflow states, labels, quality-issue titlesEpic: Component: …
GITBranches, commits, PR titlestype(KEY): …
FILESYSTEMPBI tree, spec/plan files, KATA foldersEPIC-<KEY>-<slug>

GROUP vs CONTAINER — both "contain", but GROUP only groups cases by feature (describe / Test Set), while CONTAINER is the planning & execution vessels (Plan / Run). GROUP organizes; CONTAINER plans + runs.

How the layers fit — one picture before the details

The mental model

Each layer nests in the next. A Story is planned, groups its cases, gets executed, and the code mirrors it — all tied by the same Jira key.

STORY PROJ-101 — the unit of work
 ├─ ATP: PROJ-101: … PLAN: what will be tested · CONTAINER
 ├─ TS: Validate login GROUP: groups the cases
 │  ├─ PROJ-101: TC1: should grant access… CASE
 │  └─ PROJ-101: TC2: should reject login… CASE
 ├─ ATR: PROJ-101: Story Testing RESULTS: what passed/failed · CONTAINER
 └─ @atc('PROJ-101') loginWithValidCredentials() CODE mirrors the case
same Jira key throughout Plan → Run Group contains Cases Code mirrors Case via @atc

Layer · CASE — the most important shape in the codex

Anatomy of a Test Case title

{US_ID}: TC#: should <expected outcome> [<connector> <condition>] [given <precondition>]
{US_ID} — User Story key, always TC# — sequential (TC1, TC2…) should — fixed, verb-first <outcome> — verb + object asserted [connector] — when / if · optional [given] — scope / permission · optional

EXAMPLES

PROJ-101: TC1: should grant access when credentials are valid
PROJ-101: TC2: should reject login if password is incorrect
PROJ-101: TC3: should cap input when exceeding 50 chars
PROJ-101: TC4: should block checkout when cart is empty given a logged-in user

REJECT — vague, not the word "should"

Login test — no ID, no TC#, no outcome
Login - error — too vague
TC1: Test form — missing US_ID prefix
should work correctly — no concrete outcome

Casing rule — should is lowercase because it continues the {US_ID}: TC#: prefix (a running sentence). Validate is capitalized because it starts the group summary. The layer decides the casing.

Layer · CONTAINER — execution & setup vessels

Execution & setup vessels

A vessel is the subset of artifacts that holds a run or a plan. Everything in the codex is an artifact; the vessel is the one that gets executed or planned — which is why we don't just call it an "artifact".

ATR · Acceptance Test Results (Story Testing · Test Execution issue)

ATR: {STORY-KEY}: Story Testing

STR: Sprint#{N}: Regression Testing

ReTest (bug re-test)

ReTest: {BUG-KEY}: {summary}
ReTest: GX-202: Wrong error on invalid password

Layer · JIRA workflow — the User Story / Feature lifecycle

Story workflow

NewIn progressDoneAborted / rejectedForwardReturn (back)

⚡ Global (from any status): Create → Backlog · Ready For Dev · Ready For QA · ABORTED

Happy path runs Backlog → Shift-Left QA → Estimation → Ready For Dev → In Progress → In Review → Ready For QA → In Test → QA Approved → Ready For Release → Deployed to Production.

Notes. Happy path is the left-to-right spine across both rows. Main loops: "back" returns at almost every stage, "needs quality" sends Estimation back to Shift-Left QA, and the BLOCKED branch (defect reported → Fix defect / back to dev) handles defects found in testing. Globals Create / Ready For Dev / Ready For QA / ABORTED apply from any status; Recover brings an ABORTED story back to Ready For Dev.

Layer · JIRA workflow — Bug / Defect / Improvement (one shared lifecycle)

Bug / Defect / Improvement

NewIn progressDoneAborted / rejectedForwardReturn (back)

⚡ Global (from any status): Create → Open · Re-Open → Open · ABORTED

Happy path runs Open → In Progress → In Review → Ready For QA → Closed; from Open, triage forks to CNR / Deferred / Duplicated / REJECTED / Enhancement.

Notes. Bug, Defect and Improvement share ONE byte-identical workflow. Happy path is the top spine. Triage fans out of Open (is CNR / defer / is duplicated / is WAD / is not a Bug); Deferred can resume fixing. Loops: Hard pushed skips review, "back" reopens a Closed item for re-test, and Re-Open (global) reopens any resolved item.

Layer · JIRA workflow — the Test Case lifecycle

Test workflow

NewIn progressDoneAborted / rejectedForwardReturn (back)

⚡ Global (from any status): Create → Draft · Deprecated → DEPRECATED

Automation path runs Draft → In Design → READY → In Review → Candidate → In Automation → Pull Request → AUTOMATED; the MANUAL branch can re-enter automation.

Notes. Automation happy path is the left-to-right spine across both rows. MANUAL is the hub: a case can be sent for manual (from READY / Candidate / AUTOMATED) and re-enter automation (automated / for automation / automation review). Loops: "back" steps down the chain at every stage, Fix sends AUTOMATED back to Pull Request. Deprecated (global) retires any case; recover brings DEPRECATED back to Draft.

The real laws — every name in the codex obeys these five

The Five Axioms

Not loose rules: they are the pattern that emanates from the whole nomenclature. If a new name does not fit these five, it is malformed.

A · KEYEvery name carries its Jira key — so Story ↔ test ↔ code ↔ branch stay tiedPROJ-101: …
B · LAYERThe first token encodes the layer/altitude — read before the sentenceATP: · TS: · @atc
C · OUTCOMECases are named by their outcome, not their action — verb firstshould <outcome>
D · PLAN vs RESULTSPlan and results are distinguished in the first token (P = Plan, R = Results)ATP → ATR
E · ONE SHAPEOne shape per artifact; membership is a link, never stuffed into the namelink ≠ prefix

The next slide applies these axioms to the 5 mistakes most often made by ignoring them.

Applying the axioms — the confusions to never make, as rules

Mistakes & Golden Rules

1 · should = case · Validate = group

A case asserts a behavior (should); the group only names the box (Validate). Never swap them.

2 · ATC = Acceptance Test Case

Not "Automated". ATC is the KATA method mapped 1:1 to a TC; the @atc key ties it to Jira.

3 · R = Results, not Run

ATR / FTR / STR are Test Results. A "run" is just one iteration of the execution; Results is the umbrella.

4 · Prefix = the Story key

A TC title is prefixed by its User Story key. Test Set membership is a link, never baked into the name.

5 · @atc('PROJ-101') literal

The decorator carries a literal Jira-key string — no template literals, no synthetic IDs.

6 · @integration, not @api

The API-test tag is @integration (counterpart to @e2e). @api/ is the import alias — a different thing.

7 · Casing follows the layer

should lowercase (continues the prefix) · Validate capitalized (starts the group summary).

8 · One shape · first token = layer

One canonical shape per artifact; the first token encodes its layer/altitude. Read intent before the sentence.
