CRITICAL FEAT-WS-004

Aislamiento de inquilinos (RLS)

Supabase Row-Level Security es el único mecanismo de autorización. Una sola policy faltante expone datos entre workspaces sin alerta, sin log, sin UI.

¿Qué suele romperse?

  • Endpoints PostgREST añadidos sin RLS
  • SECURITY DEFINER con recursión infinita
  • Bulk PATCH que salta policy checks
  • SELECT con IDs manipulados

Qué revisar

  • Create en workspace A → read/update/delete desde B → 403
  • PAT token no debe listar recursos de otro workspace
  • current_workspace_id() no retorna ID incorrecto en session switch
  • Flag ?include_archived no bypasses RLS

Riesgo: Falla silenciosa — no hay UI, log, ni alerta. Solo visible con pruebas activas.

CRITICAL FEAT-ATC-001

Integridad del guardado de ATC

El RPC bunkai_save_atc es la única escritura atómica que persiste header + steps + assertions + AC bindings. Si falla — network drop, constraint violation, version collision — el ATC completo se pierde.

Casos de prueba clave

  • Crear ATC con 5 steps, 3 assertions, 2 AC anchors → verificar GET
  • Editar title → verificar version bump
  • Save concurrente desde 2 tabs → segundo falla (version conflict)
  • Remover todos los steps → save → empty steps persistidos
  • YAML corrupto en assertions → 422 con error claro

Dependencias

  • ATC CRUD (FEAT-ATC-001)
  • ATC anchoring (FEAT-ATC-002)
  • Step/assertion parsing (FEAT-ATC-003)

Lo que NO se prueba hoy

  • Edit propagation a tests (no existen)
  • History table (no existe)

CRITICAL FEAT-AUTH-002

Autenticación de sesión

El edge middleware de sesión es la puerta de entrada a todas las páginas autenticadas. Un bypass o cookie expirada sin refresh silencioso = acceso no autorizado.

Flujo

Session cookie │ ▼ Edge middleware ── válida ──► página renderizada │ ✗ inválida │ ▼ Supabase auto-refresh │ ├── éxito ──► cookie renovada, request continúa └── fallo ──► redirect a login

Qué revisar

  • Cookie expirada → auto-refresh funciona?
  • Session de otro workspace → redirigir o 403
  • Middleware no deja pasar requests sin session
  • Logout → cookie invalidada

HIGH FEAT-AUTH-005 · FEAT-API-006

PAT auth + Bearer middleware: dead code

requireBearerToken() y requireScope() existen en lib/api/middleware/bearer.ts pero ninguna ruta los llama. Regresiones confirmadas en staging (BK-84, BK-135).

Hallazgo crítico

// middleware definido pero NO usado:
export function requireBearerToken() { ... }
export function requireScope() { ... }

// ninguna ruta: router.use(requireBearerToken)

PAT auth no puede verificarse end-to-end vía API — solo CRUD de tokens

Qué SÍ es testeable

  • POST /api/v1/tokens → bk_pat_ prefix + SHA-256 hash
  • DELETE /api/v1/tokens/{id} → soft-revoke, verify no listable
  • PAT con scope workspace:member → admin op → 403
  • bk_pat_ prefijo + secret mostrado solo una vez
  • TTL enforcement: 1-day PAT, verify expired

HIGH FEAT-ATC-002 · FEAT-PROJ-002

Anclaje de ATC + Árbol de módulos

ATC anchoring (M:N)

Binding via atc_acceptance_criteria. Si se pierde en re-save, la cadena US → AC → ATC se rompe.

  • ATC → AC-1 save → 1 row in join table
  • Editar, añadir AC-2 → 2 rows (old preserved)
  • Remover AC-1 → 1 row (AC-2 only)
  • Save sin AC anchor → rejected

Module tree (self-referential)

parent_id FK + depth max 6 + materialized path via trigger.

  • Depth 1 → 6 → success, tree renders
  • Depth 7 → rejected
  • Cycle detection (A→B→A) → rejected
  • Soft-delete parent → children archived

⚠️ No hay UI de creación: modules se crean via PostgREST directo. ATC sin story+AC previos no puede crearse — bootstrap dependency manual.

Edge cases

Casos borde que developers suelen olvidar

Concurrencia

  • 2 users saving same ATC → version collision
  • 2 users creating same project slug (workspace-scoped)
  • Revoke PAT while in-flight request

Permisos (FEAT-WS-002)

  • Member cannot delete workspace
  • Viewer cannot create/edit/delete
  • PAT inherits role at creation — role changes must NOT affect existing tokens (BK-135)

Data limits

  • Module depth >6 → rejected
  • Description >50 KB → rejected (BK-099)
  • PAT TTL up to 365 days → expiry enforcement

Estados huérfanos

  • Delete user → qué pasa con sus ATCs?
  • Archive module → cascade a children
  • Delete project → cascade a modules/US/AC/ATC?

ATC save edge cases

  • Empty steps → save ok
  • Same AC anchored twice → dedup
  • Steps markdown with code blocks → round-trip parse/serialize

Auth edge cases

  • Session cookie expired → auto-refresh?
  • PAT malformed hash → 401, not 500
  • Token from otro workspace scope → 403

Pre-release checklist

12 items testeables hoy

Priority-ordered. System boundary = ATC authoring.

  1. Verificar RLS en toda entidad con workspace_id
  2. Setup chain: sign-up → workspace → project → module → US → AC → ATC
  3. Crear ATC con 5 steps, 3 assertions, 2 AC anchors
  4. Editar ATC title → version incrementado
  5. Save concurrente desde 2 tabs → version conflict
  6. Module depth 7 → rejected; depth 6 → success
  7. Soft-delete module → children archivados
  8. Crear PAT, listar, revocar, verify no listado
  9. PAT member scope → admin op → 403 (BK-135)
  10. Entity workspace A → workspace B no accede (RLS)
  11. API error envelope: invalid input → 422 con { error }
  12. ATC save sin AC anchor → rejected

Phase 2+ (not testable yet)

  1. Edit ATC referenced by 2+ tests → propagate
  2. Run state machine: pass/fail/abort
  3. Idempotency: POST /runs with key → 201; same key → 200
  4. Import 501 Jira issues → 500 created, 501st rejected
  5. File bug from run → auto-context + Jira sync
  6. Rate-limit burst: 101 writes/min → 429

Discovery gaps

Hallazgos de la exploración

CRUD coverage críticamente baja

Solo atcs y access_tokens tienen CRUD completo via API versionada. El resto son read-only via PostgREST. Runs engine, tests, bugs, activity_log no migrados.

Bearer middleware = dead code

requireBearerToken() existe pero cero rutas lo usan. PAT auth sin enforce end-to-end.

OpenAPI spec incompleta (~20%)

Solo 5 endpoints documentados (Health, Auth, Tokens). PostgREST endpoints, Server Actions, UI data layer ausentes.

No hay UI para creación de entidades

Projects, modules, US, ACs no tienen formulario de creación. Se crean via imports externos o DB directo.

Idempotency no funcional

Header validation only — no replay store. Duplicate POSTs no detectados.

SLAs de integraciones no documentados

Rate limits y timeouts de Supabase Auth, PostgREST, OAuth providers no están documentados.
